# Kocotree Skills 需求访谈决策

本文档记录已经确认的产品问题、选择和最终规则。具体接口字段以 [`openapi.yaml`](./openapi.yaml) 为准。

## 1. 产品与导航

### 1.1 产品形态

**问题：** Skill 是单机管理还是共享平台？

**选择：** 桌面客户端连接共享服务，任何人可以浏览、安装、上传和更新。

**最终规则：** 当前先完成客户端、契约和模拟数据；真实后端可以从开发电脑迁移到共享服务器。

### 1.2 左侧导航

**问题：** “我的 Skill”放在账户菜单还是主导航？

**选择：** 左侧独立入口。

**最终规则：** 左侧包含 Skill 浏览、我的 Skill、上传 Skill；我的 Skill 在领域上区分本地与在线，页面布局在实现阶段确定。

### 1.3 排行榜

**问题：** 是否立即增加按下载量和上传量排序的排行榜？

**选择：** 暂缓。

**最终规则：** 排行榜不进入当前首版页面与接口。

## 2. 用户与认证

### 2.1 浏览和登录

**问题：** 哪些操作需要登录？

**选择：** 浏览匿名开放，安装和所有写操作需要飞书登录。

**最终规则：** 登录完成后恢复被拦截的动作并重新确认，不要求用户重新点击。

### 2.2 用户资料

**问题：** 姓名、头像和部门由谁维护？

**选择：** 完全以飞书为准，平台只读。

**最终规则：** 保存飞书 `openId`、姓名、头像、部门路径、状态和同步时间；部门路径是层级数组，例如 `["运营中心", "运营办"]`。

### 2.3 停用账号

**问题：** 离职或飞书账号失效后如何展示？

**选择：** 保留最后同步资料并标记停用。

**最终规则：** 停用用户不能登录或写入，历史 Owner、协作者和版本记录继续显示姓名、头像、部门与“已停用”。

### 2.4 管理员来源

**问题：** 谁可以成为平台管理员？

**选择：** 服务端配置飞书账号列表。

**最终规则：** 配置保存飞书 `openId`；客户端和普通用户不能授予管理员权限。

## 3. Owner 与协作者

### 3.1 身份划分

**问题：** 是否需要 Creator、Owner、协作者、上传者等多个角色？

**选择：** 保留 Owner、协作者和版本上传者，不单独保存 Creator。

**最终规则：** 首个版本上传者可推导原始创建者并成为初始 Owner；每个版本保存实际 `uploadedBy`；成功发布过版本的人属于协作者。

### 3.2 开放更新

**问题：** 非 Owner 能否直接更新 Skill？

**选择：** 任何已登录用户都可以直接发布新版本。

**最终规则：** 发布成功后成为协作者；发布失败、冲突或内容未变化不产生协作者身份。

### 3.3 协作者的永久性

**问题：** 协作者的版本全部撤回后是否仍算协作者？

**选择：** 仍然保留。

**最终规则：** 协作者是永久贡献记录，不能由 Owner 或管理员删除。

### 3.4 前 Owner 与协作者

**问题：** 未发布过版本的前 Owner 是否算协作者？

**选择：** 不算。

**最终规则：** 历任 Owner 从所有权审计查询；协作者只从成功版本聚合。

### 3.5 卡片和详情人员展示

**问题：** 卡片显示 Owner 还是最近上传者？详情如何显示协作者？

**选择：** 卡片显示当前 Owner，详情显示协作者头像。

**最终规则：** 协作者按中文姓名排序，默认显示前 8 个头像，其余收为 `+N`；悬浮或聚焦时显示姓名和停用状态。

## 4. 所有权转移

### 4.1 正常转移

**问题：** Owner 能否直接把所有权转给其他人？

**选择：** 只能邀请已有协作者，并由接收人确认。

**最终规则：** 邀请有效期 7 天；同一 Skill 只能有一条待确认邀请；发起人可取消，接收人可拒绝。

### 4.2 管理员权限边界

**问题：** Owner 有效时管理员能否强制转移？

**选择：** 不能。

**最终规则：** 有效 Owner 的正常转移只能由 Owner 发起。

### 4.3 停用 Owner

**问题：** Owner 停用后怎么转移？

**选择：** 管理员向协作者发起代转移，接收人仍需确认。

**最终规则：** 转移完成前显示“已停用·待转移”；协作者可以继续发布版本，Owner 专属展示名称保持冻结。

### 4.4 无协作者的停用 Owner

**问题：** 停用 Owner 的 Skill 没有协作者怎么办？

**选择：** 管理员可以邀请任意已登录过平台的活跃用户接管。

**最终规则：** 候选人仍需确认；管理员必须填写原因并保留审计记录。

## 5. 平台展示信息

### 5.1 字段来源

**问题：** Skill 原始名称、原始描述和平台文案如何区分？

**选择：** `skillName`、`skillDescription` 来自 `SKILL.md`，平台另存 `displayName`、`displayDescription`。

**最终规则：** 原始 ZIP 不改写；创建时两个展示字段都必填，并默认从原始字段填充。

### 5.2 修改权限

**问题：** Owner 和协作者能修改哪些展示字段？

**选择：** `displayName` 仅 Owner 可改，展示简介和 Tags 允许 Owner 与协作者修改。

**最终规则：** 已有协作者可以单独修改；首次更新者可以在版本事务成功后成为协作者并同时应用展示简介和 Tags。

### 5.3 原始描述同步

**问题：** 新版本修改 `skillDescription` 后是否同步展示简介？

**选择：** 发布者未明确填写展示简介时自动同步。

**最终规则：** 明确提交的 `displayDescription` 优先；自动同步和手工修改都写入审计记录。

### 5.4 并发覆盖

**问题：** 多个窗口同时修改平台展示信息怎么办？

**选择：** 后提交覆盖前提交。

**最终规则：** 不增加元数据版本锁，保留每次 `updatedBy`、时间和变更字段审计。

### 5.5 展示名称重名

**问题：** `displayName` 是否需要唯一？

**选择：** 允许重名，但必须提醒并确认。

**最终规则：** 提示中展示同名 Skill 的 `skillName`、Owner 和简介；真正唯一身份仍是 `skillName`。

## 6. Skill 名称

### 6.1 格式

**问题：** 不规范 `skillName` 是自动改写还是拒绝？

**选择：** 拒绝并给出建议，不修改 ZIP。

**最终规则：** 名称最长 64 字符，匹配 `^[a-z0-9]+(?:-[a-z0-9]+)*$`，按不区分大小写方式保证全局唯一。

### 6.2 保留名称

**问题：** 与 Claude、Codex 或平台内置名称冲突怎么办？

**选择：** 平台维护配置化保留名称列表并禁止创建。

**最终规则：** 已发布名称后来成为保留名称时进入 `NAME_CONFLICT`，继续公开展示但禁止安装和更新；本地继续使用，并推荐使用新名称创建派生 Skill。

## 7. 版本

### 7.1 版本号

**问题：** 版本号手工填写还是自动生成？

**选择：** 手工填写，更新时默认在当前最高历史版本上增加 `0.0.1`。

**最终规则：** 新 Skill 与派生 Skill 固定从 `1.0.0` 开始；版本必须是更高 SemVer，已撤回版本号仍永久占用。

### 7.2 更新说明

**问题：** 更新说明是否必填、发布后能否修改？

**选择：** `1.0.0` 默认“首次发布”，后续版本必填，发布后不可修改。

**最终规则：** 需要修订内容时发布更高版本；历史说明保持不可变。

### 7.3 内容未变化

**问题：** ZIP 内容不变但版本号或说明变化能否发布？

**选择：** 不允许。

**最终规则：** `contentHash` 与任一历史版本一致时返回 `CONTENT_UNCHANGED`。

### 7.4 并发发布

**问题：** 两人同时基于同一版本发布怎么办？

**选择：** 第二个请求因 `baseVersionId` 过期而拒绝。

**最终规则：** 平台不自动改版本号或覆盖；用户查看最新版后重新制作，或创建派生 Skill。

### 7.5 历史版本继续发展

**问题：** 明确基于历史版本修改能否发布到原 Skill？

**选择：** 必须创建派生 Skill。

**最终规则：** 原 Skill 保持单一线性版本历史。

### 7.6 版本撤回

**问题：** 哪些版本可以撤回，撤回后如何处理？

**选择：** `1.0.0` 永远不可撤回；其他版本填写原因后可撤回且不可恢复。

**最终规则：** 版本上传者可撤回自己的版本，Owner 和管理员可撤回任意非首版；最新版撤回后自动回退到最高的 `PUBLISHED` 版本。

### 7.7 已安装撤回版本

**问题：** 客户端是否自动回退？

**选择：** 显示风险和原因，由用户确认。

**最终规则：** 用户可以暂时继续使用；切换前检查本地修改并备份，不自动操作文件系统。

## 8. 派生 Skill

### 8.1 派生身份

**问题：** 不同需求是否使用同一 Skill 的复杂版本分支？

**选择：** 创建新 `skillName`，记录派生来源。

**最终规则：** 保存 `forkedFromSkillId` 和 `forkedFromVersionId`；派生 Skill 从 `1.0.0` 开始。

### 8.2 派生所有权

**问题：** 原 Owner 是否管理派生 Skill？

**选择：** 派生创建者成为新 Owner，原 Owner没有管理权限。

### 8.3 多级派生

**问题：** 能否从派生 Skill 再派生？

**选择：** 允许。

**最终规则：** 每个 Skill 保存直接来源，平台计算完整来源链。

### 8.4 来源展示

**问题：** 原 Skill 与派生 Skill 是否双向展示？

**选择：** 派生 Skill 显示来源，原 Skill 不显示派生列表。

**最终规则：** 来源归档时保留来源 `skillName` 和版本号，显示归档状态但不提供公开详情跳转。

### 8.5 回流

**问题：** 派生内容能否自动合并回原 Skill？

**选择：** 平台不提供回流或合并功能。

**最终规则：** 用户可以自己制作使用原 `skillName`、基于原当前版本的 ZIP，并按普通更新发布。

### 8.6 Agent 触发冲突

**问题：** 派生 Skill 改名是否影响 Claude/Codex 触发？

**选择：** 新名称改变手动调用身份，`description` 决定主要自动触发范围。

**最终规则：** 同一来源链同时安装时提示可能重叠，推荐只保留一个但不强制；首版以派生链判定相似，不做语义 AI 判断。

### 8.7 一键替换

**问题：** 用户选择只保留派生 Skill 时怎么处理？

**选择：** 安装验证新 Skill 后备份并移除旧 Skill，失败自动恢复。

## 9. 归档与恢复

### 9.1 归档权限

**问题：** Skill 如何下线？

**选择：** 不物理删除，由 Owner 或管理员填写原因后归档。

**最终规则：** 归档后从公开列表、搜索和详情消失，`skillName` 继续占用，本地文件照常使用。

### 9.2 已安装客户端

**问题：** 公开详情消失后如何向已安装用户解释归档？

**选择：** 按本地 `skillId` 返回最小归档状态。

**最终规则：** 只返回状态、归档时间和原因，不返回版本、下载地址或平台展示信息。

### 9.3 归档期间协作

**问题：** 谁能在归档期间上传修复版本？

**选择：** Owner、已有协作者和管理员。

**最终规则：** 上传不会自动恢复公开；只有 Owner 或管理员能填写恢复原因并恢复，且不强制先发新版本。

### 9.4 派生 Skill

**问题：** 原 Skill 归档是否影响派生 Skill？

**选择：** 不影响，派生 Skill 独立运行。

## 10. 上传与 ZIP

### 10.1 上传形式

**问题：** 接受 ZIP、文件夹还是自动打包本地 Skill？

**选择：** 只接受用户手工准备的 ZIP。

**最终规则：** 最大 50 MB，支持根目录 `SKILL.md` 或单层外包装目录；我的 Skill 不自动打包未知本地目录。

### 10.2 解析流程

**问题：** 使用服务端临时 `uploadId` 还是本地解析？

**选择：** 本地预览，发布时一次提交，服务端重新校验。

**最终规则：** 本地解析没有有效期；新建使用 `POST /api/skills`，更新使用 `POST /api/skills/{skillId}/versions`，路径不包含 `/v1`。

### 10.3 事务

**问题：** 版本成功但展示信息失败时是否保留版本？

**选择：** 整体回滚。

**最终规则：** ZIP、版本、协作者身份与请求中的展示信息全有或全无。

### 10.4 Frontmatter 兼容

**问题：** 未识别的 `SKILL.md` frontmatter 怎么处理？

**选择：** 只校验通用必填字段，其他字段原样保留。

## 11. 文件树

### 11.1 展示范围

**问题：** 文件树只显示名称还是支持内容预览？

**选择：** 文本文件可预览，二进制文件只显示名称、大小和哈希。

**最终规则：** 文件树对应具体版本，不提供单文件下载；内容按需加载的接口细节由后端实现决定。

## 12. 本地安装

### 12.1 安装目录

**问题：** Claude 与 Codex 如何共用 Skill？

**选择：** 统一安装到 `~/.agents/skills`，Claude 使用目录链接。

**最终规则：** Claude 目录非空时不移动、不覆盖，Codex 安装仍可继续。

### 12.2 产品动作

**问题：** 是否提供“只下载 ZIP”？

**选择：** 首版只提供“安装”。

### 12.3 安装来源

**问题：** 如何判断平台安装和本地 Skill？

**选择：** 外部安装凭证与内容哈希结合。

**最终规则：** 凭证保存在 `~/.agents/.kocotree/`；凭证缺失时以 `skillName + contentHash` 恢复。匹配只能证明内容一致，不能证明最初的安装动作来源。

### 12.4 同名冲突

**问题：** 本地已有同名不同内容时怎么办？

**选择：** 提示冲突并提供“备份并强制安装”。

**最终规则：** 先校验目标包、备份原目录、原子替换并在失败时恢复；每个 Skill 保留最近 3 份备份。

### 12.5 完全相同内容

**问题：** 本地内容与平台版本完全相同时怎么办？

**选择：** 只恢复凭证。

**最终规则：** 不重新安装、不备份、不新增安装次数。

### 12.6 历史版本降级

**问题：** 已安装较新版本时能否安装历史版本？

**选择：** 允许，确认后备份并降级。

**最终规则：** 展示版本与文件变化摘要；降级后继续显示“有新版本”，但不自动升级或强提醒。

### 12.7 安装统计

**问题：** 什么情况下增加安装次数？

**选择：** 本地安装最终成功后上报。

**最终规则：** 重复安装和历史版本安装都计数；取消、失败和完全相同内容的凭证恢复不计数；事件编号保证网络重试幂等。

## 13. 通知

### 13.1 提醒强度

**问题：** 是否使用强提醒？

**选择：** 不使用。

**最终规则：** 只显示账户区域小红点，通知收进列表并提供一键全部已读；不弹窗、不发系统或飞书消息。

### 13.2 新版本通知

**问题：** 他人发布版本后通知谁？

**选择：** Owner 和已有协作者，排除操作人自己。

### 13.3 展示信息通知

**问题：** 协作者单独修改展示简介或 Tags 后通知谁？

**选择：** 只通知 Owner。

## 14. 暂缓事项

### 14.1 Tag 治理

**问题：** 谁能管理全局 Tag？

**选择：** 暂不细化。

**最终规则：** 保留选择或创建 Tag 和单个 Skill 最多 5 个的现有能力，治理规则后续设计。

### 14.2 ZIP 内容安全

**问题：** 是否限制脚本和二进制文件？

**选择：** 暂不设计文件类型白名单。

**最终规则：** 保留包结构、路径穿越、链接文件和大小校验；平台解析与安装时不主动执行内容。

### 14.3 页面细节

**问题：** 我的 Skill 的筛选、卡片和文件预览具体如何布局？

**选择：** 页面实现阶段结合实际效果确定。

**最终规则：** 文档只固定页面能力与状态，不提前锁定未验证的视觉布局。
