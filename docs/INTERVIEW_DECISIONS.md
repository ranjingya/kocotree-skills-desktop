# Kocotree Skills 需求访谈决策

本文档整理产品设计阶段已经确认的问题、选择、最终规则和原因。它用于解释设计意图；具体接口字段以 [`openapi.yaml`](./openapi.yaml) 为准。

## 1. 产品形态

### 问题

Skill 是单机管理还是团队共享？

### 选择

团队共享，每个人都可以浏览、安装、上传和发布更新。

### 最终规则

- 采用桌面客户端加共享 HTTP 服务的结构。
- 当前仓库先完成客户端设计、接口契约和模拟接口规划。
- 真实后端暂不实现，未来可以从开发电脑迁移到内部服务器。

### 原因

共享数据需要统一来源，但当前优先完成前端和契约，避免后端实现反向限制产品设计。

## 2. Skill 安装目录

### 问题

Codex 和 Claude Code 如何共用同一份 Skill？

### 选择

统一安装到 `~/.agents/skills`，Codex 直接读取，Claude 通过目录链接读取。

### 最终规则

- `~/.agents/skills` 是唯一写入目标。
- 首次安装前检测并提示创建 `~/.claude/skills` 链接。
- 不维护两份副本，避免版本不一致。

### 原因

单一内容目录更容易更新、备份和判断本地修改。

## 3. 身份认证

### 问题

首版如何识别上传和更新 Skill 的人？

### 选择

正式方案使用飞书认证；当前前端只模拟登录流程。

### 最终规则

- 匿名用户可以浏览。
- 安装、上传和发布新版本时触发登录。
- 登录完成后继续被拦截的动作。
- 左下角展示登录用户信息和状态。
- API 保留设备授权、轮询和当前用户接口。

### 原因

浏览不需要身份，写操作和安装统计需要可靠的用户归属。

## 4. 版本更新权限

### 问题

已有 Skill 允许谁发布新版本？

### 选择

任何已登录用户都可以发布新版本。

### 最终规则

- 版本不可变，每次更新创建新版本。
- 平台记录真实发布者和发布时间。
- 发布新版本不设置原上传者专属权限或管理员角色。

### 原因

团队共同维护能避免原创建者不在线时阻塞更新。

## 5. 元数据来源

### 问题

Skill 名称和描述从哪里获取，谁可以修改平台展示信息，修改后是否改写 ZIP？

### 选择

先解析 `SKILL.md`；平台展示信息由 Skill 原上传者维护，不改原始包。

### 最终规则

- `skillName` 和 `skillDescription` 来自 `SKILL.md`，并随版本保存。
- 平台独立保存 `displayName`、`displayDescription` 和 Tag。
- 只有 Skill 的 `uploadedBy` 可以修改展示名称、展示简介和 Tag。
- 平台信息修改不创建 SkillVersion，但更新 `updatedBy` 和 `updatedAt`。
- 发布新版本的请求不能修改平台信息，任意已登录用户仍可发布新版本。
- 展示名称修改不影响 Skill 名称、本地目录和版本归属。
- ZIP 与 `SKILL.md` 原样保存。

### 原因

平台需要友好的展示文案和明确的协作边界，同时必须保证下载内容与上传原件一致。

## 6. 版本编号

### 问题

版本号如何产生？

### 选择

发布者手工填写 SemVer。

### 最终规则

- 版本必须符合 SemVer。
- 更新版本必须严格高于当前最新版本。
- 相同版本号不能重复发布。

### 原因

SemVer 能表达更新规模，并与开发者常用工具保持一致。

## 7. 新版本入口

### 问题

用户从哪里开始为已有 Skill 上传新版本？

### 选择

从 Skill 详情模态框进入。

### 最终规则

- 详情模态框提供“上传新版本”。
- 上传页面绑定目标 `skillId`。
- 不通过同名自动判断，也不要求用户在上传页重新选择目标。

### 原因

用户先明确选中 Skill，可以降低更新错对象的风险。

## 8. Skill 名称不匹配

### 问题

更新 ZIP 的 `skillName` 与目标 Skill 不一致时怎么办？

### 选择

拒绝更新，并推荐发布为新 Skill。

### 最终规则

- 错误信息同时展示预期名称与实际名称。
- 用户可以返回修改 ZIP，或转入发布新 Skill 流程。
- 不允许通过更新改变聚合的 Skill 名称。

### 原因

Skill 名称是版本身份，允许修改会破坏安装目录和历史版本关联。

## 9. 上传格式

### 问题

首版接受 ZIP、文件夹还是多个文件？

### 选择

只接受 ZIP。

### 最终规则

- 最大 50 MB。
- 接受根目录 `SKILL.md` 或单层外包装目录。
- 解析与发布分成两个接口步骤。

### 原因

ZIP 在浏览器、Tauri 和未来后端中的行为一致，目录结构和校验边界清晰。

## 10. 详情页内容

### 问题

详情模态框展示完整文件树还是核心信息？

### 选择

展示基本信息、`SKILL.md`、Tag、发布者和版本历史。

### 最终规则

- 不展示完整文件树。
- 不预览 `SKILL.md` 以外的文件。
- 支持选择任意历史版本安装。

### 原因

满足浏览和版本选择需求，同时控制首版复杂度。

## 11. 历史版本

### 问题

历史版本只能查看，还是可以安装？

### 选择

允许安装任意历史版本。

### 最终规则

- 浏览页安装按钮始终指向最新版本。
- 详情模态框可以选择具体版本。
- 本地降级不修改平台最新版本。

### 原因

更新出现问题时，用户需要直接回到可用版本。

## 12. 下载与安装确认

### 问题

点击安装后是直接写入，还是先确认？

### 选择

先下载到缓存，再确认安装。

### 最终规则

- 下载后校验包哈希和内容哈希。
- 确认框展示版本、目标路径、冲突和 Claude 状态。
- 用户取消时不修改本地目录，也不上报安装事件。

### 原因

用户可以在影响文件系统前确认目标和风险。

## 13. 本地冲突和修改

### 问题

同名目录存在或内容被手工修改时怎么办？

### 选择

明确警告，备份后允许替换。

### 最终规则

- 检测到本地修改时需要二次确认。
- 完整备份当前目录后再执行原子替换。
- 安装失败自动恢复。
- 每个 Skill 保留最近 3 份备份。

### 原因

兼顾安全和可操作性，既不静默覆盖，也不永久阻止更新。

## 14. Claude 目录冲突

### 问题

`~/.claude/skills` 已经包含旧 Skill 时怎么办？

### 选择

阻止创建链接并给出说明。

### 最终规则

- 不移动、不删除、不覆盖现有目录。
- 给出手工迁移说明。
- Codex 安装仍可继续。
- 客户端显示 Claude 尚未完成配置。

### 原因

自动合并存在重名和数据丢失风险，首版优先保护已有内容。

## 15. 平台范围

### 问题

Tauri 跨平台是否意味着首版同时支持所有系统？

### 选择

首版实现 Windows，接口预留 macOS 适配。

### 最终规则

- React 页面、API 和安装流程保持平台无关。
- 文件安装、链接和权限处理放入平台安装器。
- macOS 适配器列入后续工作。

### 原因

链接、权限和打包仍有操作系统差异，先在现有 Windows 环境完成可靠实现。

## 16. 本地安装来源

### 问题

如何识别本地 Skill 是否来自平台，登记文件被删除后怎么办？

### 选择

安装凭证加内容哈希恢复。

### 最终规则

- 凭证保存在 `~/.agents/.kocotree/installations.json`。
- 不在具体 Skill 中写 `.kocotree.json`。
- 凭证丢失时，用 Skill 名称和平台版本 `contentHash` 恢复。
- 无法匹配时显示来源未知或本地已修改。

### 原因

不污染 Skill 内容，同时避免应用数据清理导致所有来源永久丢失。

## 17. Tag

### 问题

Tag 是只能选择，还是允许创建？

### 选择

可以选择已有 Tag，也可以创建新 Tag。

### 最终规则

- 每个 Skill 最多 5 个。
- Tag 只包含稳定编号和展示名称，不提供 slug。
- 去除首尾空格并按不区分大小写的规范名去重。
- 修改 Skill 的 Tag 只调整关联，不重命名全局 Tag。
- 浏览页用 Tag 替代来源筛选。
- AI Tag 建议作为未来能力，必须由用户确认，不自动写入。

### 原因

兼顾分类一致性和新增领域的灵活性。

## 18. 安装次数

### 问题

卡片数字表示下载、独立用户还是成功安装操作？

### 选择

统计所有成功安装操作。

### 最终规则

- 安装成功后由客户端上报。
- 重复安装和安装历史版本均累加。
- 下载取消、取消确认和安装失败不计数。
- 上报使用幂等事件编号，网络重试不重复增加。

### 原因

该数字准确表达完成安装的操作次数，而不是下载兴趣或独立人数。

## 19. 当前交付边界

### 问题

规划结束后立即实现哪些内容？

### 选择

当前只写设计文档。

### 最终规则

- 输出产品设计、接口参考、OpenAPI、术语表、访谈决策和 ADR。
- 不修改现有前端。
- 不实现模拟接口、真实后端或 Tauri 安装命令。

### 原因

先锁定稳定边界和字段，再开始实现，减少前后端返工。
