# Kocotree Skills 产品设计

## 1. 产品定位

Kocotree Skills 是面向 Claude Code 与 Codex 用户的桌面 Skill 浏览、发布、协作与安装工具。任何人可以匿名浏览；安装、上传、发布版本和修改平台信息时通过飞书认证。

首版客户端采用 Tauri 2、React 和 TypeScript。真实后端、真实飞书认证和 Rust 安装命令不在当前实现范围内，但页面、模拟接口和文档按最终服务端契约设计。

## 2. 首版目标

- 浏览、搜索和按 Tag 筛选在线 Skill。
- 查看平台信息、Owner、协作者、版本历史和文件树。
- 安装最新版本或任意历史版本。
- 上传 ZIP 创建 Skill 或为已有 Skill 发布版本。
- 在多人直接更新的前提下保留版本、贡献和所有权审计。
- 识别平台安装、本地修改、平台匹配和未知来源的本地 Skill。
- 安全处理同名目录、派生 Skill 重叠、降级和撤回版本。

## 3. 当前范围

### 3.1 包含

- 匿名浏览和模拟飞书登录。
- Skill 浏览、我的 Skill、上传 Skill 三个领域入口。
- 详情模态框、版本历史、文件树和文本预览。
- Skill Owner、协作者、全局管理员和所有权转移规则。
- SemVer、并发发布、撤回、归档、恢复和派生关系。
- ZIP 本地预览、一次发布和服务端可信校验契约。
- Windows 本地安装、凭证、备份、恢复和 Claude 链接设计。
- 弱提醒和待处理所有权邀请。

### 3.2 暂缓

- 排行榜及下载量、上传量排名页面。
- Tag 的全局治理、合并与停用规则。
- ZIP 脚本和二进制文件的内容安全扫描。
- 评分、评论、收藏和创作者主页。
- macOS 和 Linux 的可运行安装适配器。
- AI Tag 建议。未来只能提供用户确认后的建议，不能自动写入。

## 4. 页面与交互

### 4.1 主导航

左侧主导航包含：

1. Skill 浏览。
2. 我的 Skill。
3. 上传 Skill。

“我的 Skill”在领域上区分本地 Skill 与在线 Skill。具体页签、筛选和卡片布局在页面实现阶段确定。

左下角账户区域展示飞书头像、姓名和部门路径。点击后打开账户气泡，包含通知和退出登录；通知只通过小红点提示，并支持一键全部已读。

### 4.2 匿名浏览

1. 用户打开应用后直接进入 Skill 浏览页。
2. 列表和 Tag 请求不要求登录。
3. 搜索覆盖 `displayName`、`skillName`、`displayDescription`、`skillDescription` 和 Tag。
4. 卡片展示当前 Owner，不展示最新版本上传者。
5. 点击卡片打开详情模态框。
6. `NAME_CONFLICT` Skill 继续公开展示，但标记“名称冲突·不可安装”。

### 4.3 详情模态框

详情模态框展示：

- 展示名称、展示简介和 Tag。
- `skillName` 与当前版本的 `skillDescription`。
- 当前 Owner。
- 协作者头像组：按中文姓名排序，默认显示前 8 个，超出后显示 `+N`；悬浮或键盘聚焦展示姓名和账号状态。
- 成功安装次数。
- 版本历史、状态、上传者、更新时间、包大小和更新说明。
- 指定版本的规范化文件树。
- 文本文件按需预览；二进制文件只展示名称、大小和哈希。
- 最新版本、历史版本的安装入口，以及登录后可见的“上传新版本”。
- 派生 Skill 的直接来源和完整来源链；原 Skill 详情不反向展示派生列表。

来源 Skill 已归档时仍展示来源 `skillName`、来源版本和“来源 Skill 已归档”，不提供公开详情跳转。来源恢复后可以重新启用跳转。

### 4.4 登录拦截

以下操作要求登录：

- 安装任意版本。
- 进入上传页面。
- 创建 Skill 或发布版本。
- 修改平台展示信息。
- 处理所有权邀请。

客户端记录被拦截的动作。登录成功后重新获取目标状态，再恢复安装或上传流程并重新展示必要确认；登录失败或取消时不执行写操作。

### 4.5 发布新 Skill

1. 用户手工准备最大 50 MB 的 ZIP；平台不替本地目录自动打包。
2. 客户端在本地解析 ZIP，用于展示 `skillName`、`skillDescription`、文件树、包大小和哈希。
3. `displayName` 与 `displayDescription` 必填，默认分别使用 `skillName` 和 `skillDescription`。
4. 重名 `displayName` 需要展示同名 Skill 并要求用户确认。
5. 用户确认后通过 `POST /api/skills` 一次性提交原始 ZIP 与表单。
6. 服务端重新解析、校验并计算权威哈希，成功后创建 Skill 与固定为 `1.0.0` 的首个版本。

客户端解析状态只保存在当前页面内存中，没有 `uploadId` 和有效期。关闭或刷新页面后重新选择 ZIP。

### 4.6 发布新版本

1. 用户从目标 Skill 详情进入更新流程，页面绑定 `skillId`。
2. ZIP 中的 `skillName` 必须与目标 Skill 完全一致。
3. 页面自动把版本号预填为当前最高历史版本的下一个 Patch，用户可以手动修改。
4. 请求携带 `baseVersionId`；当前版本已变化时拒绝发布并展示冲突。
5. 更新说明必填；首个版本的默认说明为“首次发布”。
6. 内容哈希必须与该 Skill 所有历史版本不同。
7. 版本、协作者身份和可修改的平台信息在同一事务中提交。
8. 任意已登录用户可以直接发布版本；首次成功后成为协作者。

从历史版本继续发展时不能并入原 Skill，需要以新 `skillName` 创建派生 Skill。

### 4.7 平台展示信息

权限如下：

| 字段 | Owner | 协作者 |
| --- | --- | --- |
| `displayName` | 可修改 | 不可修改 |
| `displayDescription` | 可修改 | 可修改 |
| Tags | 可修改 | 可修改 |
| `skillName` | 不可修改 | 不可修改 |
| `skillDescription` | 随 ZIP 版本更新 | 随 ZIP 版本更新 |

已有协作者可以通过 `PATCH /api/skills/{skillId}` 单独修改展示简介和 Tags。首次更新者可以在版本事务成功后成为协作者，并在同一事务中应用展示简介和 Tags。

如果新版本的 `skillDescription` 发生变化且发布者未明确提交 `displayDescription`，平台自动同步展示简介。平台信息采用后提交覆盖前提交，并记录 `updatedBy`、`updatedAt` 和变更字段；不创建 `SkillVersion`。协作者单独修改后只向 Owner 发送弱提醒。

## 5. 用户、Owner 与协作

### 5.1 User

用户头像、姓名和部门完全来自飞书，平台不允许编辑。User 包含平台编号、飞书 `openId`、姓名、头像、部门路径、状态、角色和最近同步时间。

飞书资料变化后展示当前资料。账号停用时保留最后一次同步的姓名、头像和部门，并标记 `DISABLED`；停用用户不能登录或执行写操作。

管理员由服务端配置的飞书 `openId` 列表确定，普通用户和客户端不能授予管理员权限。

### 5.2 Owner 与协作者

- 首个版本上传者成为初始 Owner。
- 原始创建者从 `1.0.0` 的 `uploadedBy` 得到，不单独保存 `createdBy`。
- 任何成功发布版本的人永久成为协作者；撤回其版本不会删除贡献记录。
- 当前 Owner 在详情中单独展示，不在协作者头像组中重复。
- 未发布过版本的前 Owner 只保留在所有权历史中，不算协作者。
- 卡片展示当前 Owner；每个版本展示实际 `uploadedBy`。

### 5.3 所有权转移

- 有效 Owner 只能向已有协作者发起转移。
- 接收人必须确认；邀请 7 天后失效，发起人可以取消，接收人可以拒绝。
- 同一 Skill 同时只能存在一条待确认邀请。
- 接受后新用户成为 Owner，原 Owner 是否显示为协作者取决于其是否发布过版本。
- Owner 有效时管理员不能强制转移。
- Owner 停用时管理员可以向已有协作者发起代转移；仍需接收人确认。
- Owner 停用且没有协作者时，管理员可以选择任意已登录过平台的活跃用户；候选人仍需确认。
- 代转移必须填写原因并保留审计记录。

停用 Owner 在转移完成前显示“已停用·待转移”。协作者仍可发布内容版本，但 Owner 专属的 `displayName` 修改保持冻结。

## 6. Skill 与版本模型

### 6.1 Skill

| 字段 | 含义 |
| --- | --- |
| `id` | 平台生成的稳定标识。 |
| `skillName` | `SKILL.md` 的 `name`，全局唯一且不可修改。 |
| `displayName` | 平台展示名称，允许重复。 |
| `skillDescription` | 当前可用版本 `SKILL.md` 的 `description`。 |
| `displayDescription` | 平台展示简介。 |
| `status` | `ACTIVE`、`ARCHIVED` 或 `NAME_CONFLICT`。 |
| `owner` | 当前 Owner。 |
| `tags` | 最多 5 个 Tag。 |
| `currentVersion` | 当前最高且为 `PUBLISHED` 的版本。 |
| `updatedBy` | 最近发布版本或修改平台信息的用户。 |
| `installCount` | 成功安装操作总数。 |
| `forkedFromSkillId` | 派生 Skill 的直接来源 Skill，可空。 |
| `forkedFromVersionId` | 派生 Skill 的直接来源版本，可空。 |
| `createdAt` | 创建时间。 |
| `updatedAt` | 最近更新时间。 |

### 6.2 SkillVersion

`SkillVersion` 发布后不可修改：

| 字段 | 含义 |
| --- | --- |
| `id` | 版本稳定标识。 |
| `version` | SemVer。 |
| `status` | `PUBLISHED` 或 `WITHDRAWN`。 |
| `skillName` | 该版本 `SKILL.md` 的名称。 |
| `skillDescription` | 该版本 `SKILL.md` 的描述。 |
| `skillMd` | 原始 `SKILL.md` 快照。 |
| `fileManifest` | 规范化文件清单。 |
| `changelog` | 更新说明。 |
| `baseVersionId` | 发布时依据的版本；首个版本为空。 |
| `packageSize` | ZIP 字节数。 |
| `packageSha256` | ZIP 字节哈希。 |
| `contentHash` | 规范化目录内容哈希。 |
| `uploadedBy` | 实际上传者。 |
| `publishedAt` | 发布时间。 |
| `withdrawnBy` | 撤回人，可空。 |
| `withdrawnAt` | 撤回时间，可空。 |
| `withdrawalReason` | 必填的撤回原因，可空。 |

### 6.3 版本规则

- 新 Skill 和派生 Skill 均从 `1.0.0` 开始。
- 更新时默认增加 `0.0.1`，允许发布者手动填写更高 SemVer。
- 新版本必须高于全部历史版本，包括已撤回版本；版本号永久占用。
- 更新说明在后续版本必填并不可修改。
- `contentHash` 与任一历史版本相同则返回内容未变化错误。
- `baseVersionId` 不是当前版本时返回并发冲突。
- `1.0.0` 永远不能撤回。
- 其他版本可由该版本上传者、Owner 或管理员填写原因后撤回；撤回不可逆。
- 撤回当前版本后自动回退到最高的 `PUBLISHED` 历史版本。

本地已经安装撤回版本时，客户端显示风险、原因和推荐可用版本，由用户确认是否切换；不自动修改本地目录。

## 7. 派生 Skill

- 从历史版本独立发展时创建派生 Skill，使用新的合法 `skillName`。
- 派生 Skill 的创建者成为新 Owner，原 Skill Owner 没有管理权限。
- 派生 Skill 从 `1.0.0` 开始，保存直接来源 Skill 和版本。
- 允许多级派生，完整链路通过直接来源计算。
- 原 Skill 归档不影响派生 Skill。
- 平台不提供回流、自动合并或合并请求；用户可以自行制作符合原 Skill 当前版本要求的 ZIP，并作为普通更新发布。
- 原 Skill 详情不展示派生列表。

安装派生 Skill 时，如果本地已经安装同一来源链上的 Skill，客户端提示自动触发可能重叠并推荐只保留一个。用户可以同时保留，也可以使用“一键替换”：安装并验证新 Skill 后备份并移除旧 Skill，失败时恢复。

## 8. 名称规则

`skillName` 必须匹配：

```text
^[a-z0-9]+(?:-[a-z0-9]+)*$
```

- 最长 64 个字符。
- 全局唯一比较不区分大小写。
- 不允许空格、下划线、中文或连续连字符。
- 不符合规则时拒绝发布并给出建议名称；平台不修改 ZIP。
- 平台从配置读取 Claude、Codex 和平台保留名称；命中后禁止创建。
- 已发布名称后来成为保留名称时，Skill 进入 `NAME_CONFLICT`：继续公开展示，但禁止安装和发布版本；本地文件继续使用，并推荐使用新名称创建派生 Skill。

`displayName` 允许重复。创建或修改时发现重名需要展示同名 Skill 的 `skillName`、Owner 和简介，由用户确认后继续。

## 9. 归档与恢复

- Owner 或管理员可以归档 Skill，必须填写不可修改的归档原因。
- 归档后公开列表、搜索和详情统一返回不存在。
- Owner、已有协作者和管理员仍可从在线管理记录查看完整信息并上传版本。
- 只有 Owner 或管理员可以恢复公开；恢复必须填写原因，不强制发布新版本。
- 已安装客户端按本地 `skillId` 只能查询到 `ARCHIVED`、归档时间和归档原因，不返回版本、下载地址或展示信息。
- 本地文件继续可用，不自动删除。

## 10. ZIP 与内容校验

### 10.1 接受结构

只接受最大 50 MB 的 ZIP，允许根目录直接包含 `SKILL.md`，或使用一个单层外包装目录。单层外包装目录在文件清单、哈希和安装路径中被移除。

`SKILL.md` 必须包含合法的 `name` 和 `description`。其他 frontmatter 字段原样保留，不写入核心 Skill 实体，也不因平台暂时不认识而拒绝。

### 10.2 基础拒绝条件

- ZIP 损坏或无法读取。
- 缺少合法的 `SKILL.md`。
- 包含绝对路径、`..` 路径穿越或规范化后越界路径。
- 包含符号链接、目录联接或其他链接类型。
- 解压后超过 200 MB、普通文件超过 2,000 个或存在大小写路径冲突。

脚本和二进制内容安全扫描暂缓；平台解析和安装过程中不主动执行 ZIP 内容。

### 10.3 哈希

`packageSha256` 对原始 ZIP 字节计算 SHA-256。

`contentHash` 的统一算法：

1. 列出所有参与安装的普通文件。
2. 相对路径统一为 `/`，按 UTF-8 字节顺序排序。
3. 为每个文件计算 SHA-256。
4. 为每项生成 `相对路径\0文件大小\0文件哈希\n`。
5. 拼接后再次计算 SHA-256。

时间戳、压缩级别、ZIP 顺序、权限和空目录不参与内容哈希。

## 11. 本地安装

### 11.1 安装目录与 Claude

- 唯一安装目标是 `~/.agents/skills`。
- Codex 直接读取通用目录。
- `~/.claude/skills` 不存在或为空时提示创建到通用目录的链接。
- Claude 目录非空时不移动、不覆盖，显示手工迁移说明；Codex 安装仍可继续。

### 11.2 安装与强制安装

产品只提供“安装”，不提供单独保存 ZIP。

正常流程：下载缓存、校验 `packageSha256`、安全解析、校验 `contentHash`、确认、备份、原子替换、最终校验、写入凭证、上报成功安装事件。

同名目录已存在时：

- 内容哈希与平台版本完全一致：恢复凭证，不重新安装、不备份、不增加安装次数。
- 内容不同：展示冲突并提供“备份并强制安装”；原目录从生效位置移除，但保留最近 3 份备份。
- 安装失败：自动恢复原目录。

### 11.3 历史版本与撤回版本

- 安装历史版本属于降级，必须展示当前与目标版本、更新说明和文件变化摘要。
- 用户确认后备份并安装；降级后继续显示“有新版本”，但不弹窗强提醒。
- 本地撤回版本显示风险与原因，允许用户暂时继续使用或确认切换到当前可用版本。

### 11.4 安装来源

本地状态包括：

- `PLATFORM_INSTALLED`：凭证存在且哈希一致。
- `PLATFORM_MODIFIED`：凭证存在但哈希不同。
- `PLATFORM_MATCHED`：凭证缺失但名称与哈希匹配平台版本。
- `LOCAL_UNKNOWN`：无法匹配平台版本。
- `MISSING`：凭证存在但目录缺失。

凭证与备份保存在：

```text
~/.agents/.kocotree/
├─ installations/
└─ backups/
```

本地内容与平台版本完全匹配只能证明内容来源可关联，不能证明用户最初一定通过平台按钮安装。

## 12. 通知

- 采用应用内弱提醒：账户区域小红点、通知列表和一键全部已读。
- 不弹窗、不发送系统通知或飞书消息。
- 新版本发布后通知 Owner 和已有协作者，但不通知操作人自己。
- 协作者单独修改展示简介或 Tags 时只通知 Owner。
- 所有权邀请和状态变化进入通知列表。
- 暂不设计免打扰和复杂订阅规则。
