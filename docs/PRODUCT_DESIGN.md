# Kocotree Skills 产品设计

## 1. 产品定位

Kocotree Skills 是面向团队成员的桌面 Skill 浏览、发布与安装工具。用户可以匿名查看平台中的 Skill；需要安装、发布新 Skill 或上传新版本时，通过飞书完成身份认证。

首版客户端采用 Tauri 2、React 和 TypeScript。接口按契约优先方式设计，当前实现阶段只使用模拟接口，不包含真实后端、真实飞书认证或本地安装命令。

## 2. 目标用户与目标

目标用户是使用 Codex 或 Claude Code 的团队成员。

首版需要让用户完成以下任务：

- 搜索并按 Tag 筛选 Skill。
- 查看 Skill 的平台信息、原始 `SKILL.md` 和版本历史。
- 选择最新版本或任意历史版本进行安装。
- 上传 ZIP 并发布新的 Skill。
- 从已有 Skill 详情中上传新版本。
- 识别本地已安装、本地已修改和来源未知的 Skill。

## 3. 首版范围

### 3.1 包含

- 匿名浏览、搜索、排序与 Tag 筛选。
- 模拟飞书登录和登录用户状态展示。
- Skill 详情抽屉。
- ZIP 解析、平台展示信息确认和发布流程设计。
- SemVer 版本历史与指定版本下载。
- Windows 本地安装、备份、恢复和 Claude 链接的设计说明。
- 模拟 API、真实 API 与本地安装器之间的边界定义。

### 3.2 不包含

- 真实后端和数据库实现。
- 真实飞书 OAuth 或设备授权实现。
- Rust 本地安装命令实现。
- Skill 删除、审核、评分、评论、收藏和管理员功能。
- 创作者广场与创作者详情页。
- macOS 和 Linux 的可运行安装适配器。
- AI 自动 Tag 接口。该能力只保留为未来方向。

## 4. 页面与用户流程

### 4.1 匿名浏览

1. 用户打开应用后直接进入 Skill 浏览页。
2. 客户端请求 Skill 列表和 Tag 列表，不要求登录。
3. 用户可以搜索 Skill 名称、展示名称、Skill 描述、展示简介和 Tag。
4. 用户可以按创建时间、最新版本发布时间或安装次数排序。
5. 点击卡片打开详情抽屉。

### 4.2 详情抽屉

详情抽屉展示：

- 展示名称、展示简介和 Tag。
- `SKILL.md` 中的 Skill 名称和 Skill 描述。
- 最新版本的 `SKILL.md` 原文预览。
- 原上传者、最近更新者、创建时间和更新时间。
- 成功安装次数。
- 全部版本历史，包括版本号、更新说明、发布者、发布时间和包大小。
- 最新版本或指定历史版本的安装入口。
- 已登录用户可见的“上传新版本”入口。

首版不展示完整文件树，也不预览 `SKILL.md` 以外的文件。

### 4.3 登录拦截

匿名用户执行以下动作时触发登录：

- 安装任意版本。
- 进入上传页面。
- 从详情抽屉上传新版本。

模拟阶段使用飞书登录弹窗。登录成功后继续用户刚才被拦截的动作，而不是返回首页。侧栏左下角展示当前用户头像、姓名和登录状态，并提供退出入口。

真实后端阶段沿用设备授权流程：客户端创建临时授权会话、打开浏览器完成飞书授权、轮询领取访问令牌，再调用当前用户接口校验令牌。

### 4.4 发布新 Skill

发布采用两步流程：

1. 用户选择一个 ZIP，客户端调用解析接口。
2. 解析接口校验包结构并返回临时上传编号、Skill 名称、Skill 描述、`SKILL.md` 内容、包大小、文件数、包哈希和内容哈希。
3. 页面展示只读的 Skill 信息，并允许填写展示名称、展示简介、版本号、更新说明和 Tag。
4. 用户确认后调用创建接口。

发布规则：

- 首个版本由用户填写 SemVer，通常为 `1.0.0`。
- Skill 名称在平台全局唯一。
- 展示名称可以与其他 Skill 重复。
- 展示信息只保存在平台，不修改原始 ZIP 或 `SKILL.md`。
- 若 Skill 名称已经存在，拒绝创建并引导用户前往该 Skill 详情上传新版本。

### 4.5 上传新版本

1. 用户从 Skill 详情抽屉点击“上传新版本”。
2. 上传页面绑定当前 `skillId`，不允许用户重新选择目标 Skill。
3. ZIP 解析完成后，比较包内 Skill 名称与目标 Skill 的 Skill 名称。
4. Skill 名称不一致时拒绝发布，展示两者差异，并推荐将 ZIP 发布为新的 Skill。
5. 新版本号必须符合 SemVer、不得重复，并且严格高于平台当前最新版本。
6. 更新说明必填；发布版本的请求不能修改展示名称、展示简介或 Tag。
7. 任意已登录用户都可以发布新版本，平台记录该版本的实际上传者。

### 4.6 修改平台信息

Skill 的原上传者可以独立修改展示名称、展示简介和 Tag。平台信息修改不创建 `SkillVersion`，但会更新 Skill 的 `updatedBy` 和 `updatedAt`。其他登录用户修改时返回 `403 NOT_SKILL_OWNER`。

修改 Tag 只替换当前 Skill 与 Tag 的关联，不重命名全局 Tag，也不影响其他 Skill。发布新版本的请求不接受平台信息字段，防止绕过原上传者权限。

## 5. 领域模型

### 5.1 用户

`User` 表示飞书认证后的平台用户，包含平台用户编号、展示名、邮箱、头像和状态。上传者、版本发布者都从访问令牌解析，客户端不能提交或伪造用户编号。

### 5.2 Skill

`Skill` 表示平台中的稳定聚合根：

| 字段 | 含义 |
| --- | --- |
| `id` | 平台生成的稳定 UUID。 |
| `skillName` | 从 `SKILL.md` frontmatter 读取的名称，全局唯一且创建后不可修改。 |
| `displayName` | 平台展示名称，可编辑且允许重复。 |
| `skillDescription` | 最新版本 `SKILL.md` frontmatter 中的描述。 |
| `displayDescription` | 平台展示简介，不写回 ZIP。 |
| `tags` | 最多 5 个 Tag。 |
| `latestVersion` | 当前最高 SemVer 版本摘要。 |
| `uploadedBy` | 首次上传并创建 Skill 的用户，创建后不变。 |
| `updatedBy` | 最近修改平台信息或发布新版本的用户。 |
| `installCount` | 客户端报告的成功安装操作总数。 |
| `createdAt` | Skill 创建时间。 |
| `updatedAt` | 最近修改平台信息或发布新版本的时间。 |

### 5.3 Skill 版本

`SkillVersion` 是不可变版本记录。每次发布都会创建新记录，Skill 名称、Skill 描述、原始 `SKILL.md`、版本号、更新说明、上传者、包哈希和内容哈希在发布后不修改。版本不包含 `updatedBy`。

| 字段 | 含义 |
| --- | --- |
| `id` | 平台生成的版本 UUID。 |
| `version` | 发布者填写的 SemVer。 |
| `skillName` | 该版本 `SKILL.md` 中的名称。 |
| `skillDescription` | 该版本 `SKILL.md` 中的描述。 |
| `skillMd` | 该版本原始 `SKILL.md` 快照。 |
| `changelog` | 版本更新说明。 |
| `packageSize` | 原始 ZIP 字节数。 |
| `packageSha256` | 原始 ZIP 字节哈希。 |
| `contentHash` | 规范化目录内容哈希。 |
| `uploadedBy` | 该版本的实际上传者。 |
| `publishedAt` | 版本发布时间。 |

安装历史版本只影响本地安装状态，不会修改平台的最新版本指针。

### 5.4 Tag

Tag 只包含稳定编号和展示名称。Tag 可以从已有项中选择，也可以在创建 Skill 或修改平台信息时创建。服务端去除首尾空格并按不区分大小写的规范名去重。单个 Skill 最多绑定 5 个 Tag。

### 5.5 上传会话

`UploadInspection` 表示 ZIP 解析后的临时结果。它通过 `uploadId` 与后续创建或更新请求关联，并包含明确的过期时间。上传会话只用于发布一次，成功发布后失效。

### 5.6 本地安装记录

`LocalInstallation` 是客户端本地状态，不属于服务端 Skill DTO。它记录平台 Skill、具体版本、本地路径、内容哈希、安装时间和服务地址。

本地记录、备份和恢复信息保存在：

```text
~/.agents/.kocotree/
├─ installations.json
└─ backups/
```

## 6. ZIP 与内容校验

### 6.1 接受的结构

首版只接受不超过 50 MB 的 `.zip` 文件，允许两种结构：

```text
skill.zip
├─ SKILL.md
└─ references/
```

或：

```text
skill.zip
└─ skill-folder/
   ├─ SKILL.md
   └─ references/
```

单层外包装目录在校验和安装时被规范化，不属于 Skill 内容根路径。

### 6.2 拒绝条件

- ZIP 损坏或无法读取。
- 缺少 `SKILL.md`。
- `SKILL.md` frontmatter 缺少合法的 `name` 或 `description`。
- 包含绝对路径、`..` 路径穿越或超出内容根目录的路径。
- 包含符号链接、目录联接或其他链接类型。
- 解压后超过 200 MB 或普通文件超过 2,000 个。
- 内部相对路径重复，或在大小写不敏感文件系统上发生路径冲突。

系统垃圾文件如 `.DS_Store`、`__MACOSX`、`._*`、`Thumbs.db` 和 `desktop.ini` 不参与安装和内容哈希。

### 6.3 哈希规则

`packageSha256` 是原始 ZIP 字节的 SHA-256。

`contentHash` 用于比较解压后的实际目录：

1. 递归列出所有参与安装的普通文件。
2. 相对路径统一使用 `/`，并按 UTF-8 字节顺序排序。
3. 为每个文件计算内容 SHA-256。
4. 为每项生成 `相对路径\0文件大小\0文件哈希\n`。
5. 拼接全部条目并计算最终 SHA-256。

时间戳、ZIP 顺序、压缩级别、空目录和文件权限不参与计算。文件字节不做编码或换行转换。

## 7. 本地安装设计

### 7.1 平台范围

首版实现目标是 Windows。安装能力通过统一接口隔离平台差异，未来增加 macOS 适配器；React 页面、API 契约和安装流程不依赖具体操作系统。

### 7.2 安装流程

1. 获取指定版本的临时下载地址。
2. 下载到应用缓存并校验 `packageSha256`。
3. 安全解析 ZIP，校验 `contentHash`。
4. 显示版本、目标目录、Claude 状态和本地冲突的安装确认。
5. 若目标目录存在，检查本地内容是否被修改。
6. 用户确认后创建完整备份。
7. 将新内容写入临时目录，再通过重命名完成原子替换。
8. 重新计算内容哈希并写入安装凭证。
9. 向平台上报成功安装事件。

下载取消、用户取消确认、哈希不匹配或安装失败均不上报安装事件。

### 7.3 冲突与恢复

- 目标目录不存在时直接安装。
- 目标目录存在且内容未修改时，确认后备份并替换。
- 内容哈希不一致时展示强警告；用户再次确认后备份本地修改并替换。
- 安装失败时自动恢复本次备份。
- 每个 Skill 最多保留最近 3 份备份，生成第 4 份后删除最旧的一份。

### 7.4 安装来源恢复

应用启动时扫描 `~/.agents/skills`：

- 有安装凭证且哈希一致：平台已安装。
- 有安装凭证但哈希不同：本地已修改。
- 无安装凭证但 Skill 名称和内容哈希匹配平台历史版本：恢复安装凭证。
- Skill 名称匹配但没有版本哈希匹配：本地存在不同内容或已修改。
- 平台不存在该 Skill 名称：来源未知。
- 安装凭证存在但目录缺失：安装目录已被外部删除。

### 7.5 Claude 链接

Codex 直接读取 `~/.agents/skills`。首次安装前检测 `~/.claude/skills`：

- 路径不存在或为空：提示用户创建指向 `~/.agents/skills` 的 Windows 目录链接。
- 路径已经是正确链接：继续安装。
- 路径存在且包含内容：不移动、不删除、不覆盖，阻止创建链接并给出手工迁移说明。

Claude 链接失败不阻止 Skill 安装到通用目录，但客户端需要显示 Claude 尚未完成配置。

## 8. 状态与失败反馈

客户端需要覆盖以下状态：

- 列表加载、空列表、请求失败和重试。
- 匿名、等待授权、已登录、授权过期和退出登录。
- ZIP 解析中、解析失败、等待确认、发布中和发布成功。
- 未安装、平台已安装、本地已修改、来源未知、目录丢失和存在更新。
- 下载中、等待确认、备份中、安装中、恢复中、成功和失败。

错误信息必须说明失败原因和下一步操作，不能只显示“操作失败”。敏感令牌不得写入普通日志。
