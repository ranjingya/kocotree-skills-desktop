# ADR-0004：Skill 所有权、版本线与派生关系

- 状态：已接受
- 日期：2026-07-17

## 背景

平台允许任何登录用户直接更新 Skill，同时需要明确当前管理责任、永久贡献归属、并发发布、历史版本分化和离职账号处理。如果把创建者、Owner、版本上传者和协作者混成一个字段，权限与审计会随版本增长而失真。

## 决策

- Skill 保存当前 `owner`；首个版本上传者成为初始 Owner，原始创建者从 `1.0.0` 的 `uploadedBy` 推导。
- 协作者由成功发布过版本的用户聚合，是永久贡献记录；未发布版本的历任 Owner 不算协作者。
- 任何登录用户可以发布版本，成功后成为协作者。
- 版本不可变，使用 `baseVersionId` 检查并发；内容哈希必须与全部历史版本不同。
- 新 Skill 与派生 Skill 从 `1.0.0` 开始；`1.0.0` 不允许撤回。
- 从历史版本独立发展时创建新 `skillName` 的派生 Skill，并保存直接来源 Skill 与版本。
- 允许多级派生，不提供自动回流或合并功能。
- 正常所有权转移仅面向已有协作者并需要接收确认；停用 Owner 由管理员按受限规则代为处理。
- Skill 不物理删除；Owner 或管理员填写原因后归档，已安装客户端只能查询最小归档状态。

## 结果

### 正面影响

- 当前管理责任、历史贡献和每个版本的实际发布者可以分别表达。
- 开放更新不会牺牲审计和并发安全。
- 不同需求使用独立 Skill，避免把同一名称变成难以解释的分支图。
- 停用账号不会删除历史，也不会永久阻塞维护。

### 代价

- 需要所有权邀请、审计记录、版本状态和派生来源等额外模型。
- 客户端需要展示 Owner、协作者、版本上传者和来源链的不同含义。
- 平台只能提示派生 Skill 的触发重叠，不能可靠判断手工复制后的语义来源。

## 后续约束

- 任何权限判断都使用当前 Owner、协作者聚合和管理员配置，不能使用含义不清的“创建者权限”。
- 版本号必须高于全部历史版本，包括已撤回版本。
- 归档、恢复、撤回和所有权转移必须记录操作人、时间与原因。
- 原 Skill 的生命周期变化不能级联修改派生 Skill。
