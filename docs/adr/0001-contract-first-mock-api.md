# ADR-0001：契约优先与模拟接口

- 状态：已接受
- 日期：2026-07-15

## 背景

客户端页面需要先行开发，真实后端和飞书认证尚未进入实现阶段。如果页面直接读取静态数组，未来接入 HTTP 时会把数据来源、加载状态和错误处理散落到组件中。

## 决策

- 使用 OpenAPI 3.1 作为服务端 DTO 的权威契约。
- 接口路径直接使用 `/api/...`，不增加 `/v1` 中间层。
- 提供人类可读的中文 API 文档，并与 OpenAPI 一一对应。
- 前端实现阶段定义统一 `SkillApi` 接口。
- `MockSkillApi` 和未来的 `HttpSkillApi` 实现同一接口。
- HTTP 响应统一使用 `code`、`data`、`msg` 外层；`HttpSkillApi` 校验业务码后向 React 组件返回 `data`。
- React 组件通过 `SkillApi` 访问服务端数据，不能直接导入模拟数据文件。
- ZIP 预解析由客户端本地解析器负责；发布时 `SkillApi` 一次性接收 ZIP 和表单字段，并模拟服务端重新校验。
- 新建 Skill 与发布版本使用独立接口；不使用临时 `uploadId`，也不依赖客户端计算的哈希作为服务端事实。
- 版本、协作者身份和同一请求中的平台展示信息采用单一事务语义。
- TypeScript 服务端 DTO 从 OpenAPI 生成，本地安装模型和本地 ZIP 解析状态手工维护并明确隔离。
- OpenAPI 描述 Skill 领域接口；登录协议和令牌生命周期由认证接入方定义，前端只依赖身份适配器和 Bearer Token。

## 结果

### 正面影响

- 前端可以独立开发加载、错误和发布流程。
- 接入后端时主要替换适配器，不重写页面。
- 字段、可空性和错误码可以自动校验。

### 代价

- OpenAPI、中文说明和模拟实现需要保持一致。
- 模拟接口不能证明真实后端的性能和存储行为。

## 后续约束

- 新接口先修改 OpenAPI，再更新说明和实现。
- CI 需要校验 OpenAPI 并检查生成类型是否有未提交变化。
- 模拟接口必须能覆盖成功、空数据、延迟和失败状态。
